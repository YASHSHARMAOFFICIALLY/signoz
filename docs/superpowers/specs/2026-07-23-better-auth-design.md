# Better Auth ‚Äî backend serves, frontend consumes

**Date:** 2026-07-23
**Branch:** `feat/better-auth`

## Goal

Stand up authentication for the signoz monorepo: `apps/backend` (Bun) hosts the
Better Auth server backed by Postgres via Prisma; `apps/frontend` (Next.js)
consumes it with the Better Auth React client. Built in learning mode ‚Äî the user
writes the pieces with real decisions, Claude scaffolds boilerplate and reviews.

## Decisions

| Item | Choice | Why |
|---|---|---|
| Auth library | Better Auth | Requested |
| Database | Postgres on Neon (cloud) | No local server; instant `DATABASE_URL` |
| ORM | Prisma | Requested; schema DSL + generated client |
| Methods | Email+password, Google + GitHub OAuth | Requested |
| Topology | Backend serves `/api/auth/*`; frontend consumes | Real two-service shape |
| Build order | Backend end-to-end (curl-tested) then frontend | Understand the engine before the UI |

## Topology

```
apps/backend  (Bun.serve)          apps/frontend (Next.js)
  Better Auth instance               better-auth/react client
  Prisma  --->  Postgres (Neon)      calls  http://backend/api/auth/*
  /api/auth/*  <---------- CORS -----------  signIn / signUp / useSession
```

## Backend (`apps/backend`)

- `prisma/schema.prisma` ‚Äî Better Auth required models: `User`, `Session`,
  `Account`, `Verification`. Generated by Better Auth CLI, pushed with
  `prisma migrate`.
- `lib/auth.ts` ‚Äî the Better Auth instance: `prismaAdapter`,
  `emailAndPassword`, `socialProviders: { google, github }`.
- `index.ts` ‚Äî `Bun.serve` routing `/api/auth/*` to `auth.handler`, CORS for
  the frontend origin.
- `.env` ‚Äî `DATABASE_URL`, `BETTER_AUTH_SECRET`, OAuth client IDs/secrets.

## Frontend (`apps/frontend`)

- `src/lib/auth-client.ts` ‚Äî `createAuthClient({ baseURL })`.
- Login/signup form component wired to `authClient.signUp.email` /
  `.signIn.email` + Google/GitHub buttons.

## Data flow (email signup)

`form -> authClient.signUp.email() -> POST /api/auth/sign-up/email -> Better
Auth hashes password, Prisma writes User+Account -> Session cookie set ->
frontend useSession() reflects it`

## Build phases (learning mode)

Each phase: Claude scaffolds boilerplate + context; user writes the meaningful
lines; Claude reviews and both verify.

1. **DB + schema** ‚Äî provision Neon, wire Prisma. User runs Better Auth
   schema-gen + `prisma migrate`, confirms tables in Neon.
2. **`lib/auth.ts`** ‚Äî Claude stubs file/imports. User writes the Better Auth
   instance config (adapter, providers, options).
3. **`index.ts` Bun.serve** ‚Äî Claude stubs server. User writes request routing
   + CORS.
4. **curl test** ‚Äî user hits signup/login, confirms rows in Neon.
5. **Frontend** ‚Äî auth-client + form; user writes the form submit logic.

## Out of scope (YAGNI)

- Email verification / magic links (no email sender wired) ‚Äî add later.
- The 7 empty backend scaffold dirs' `src/` vs flat layout question ‚Äî decided
  as files land, not upfront.
- Role/permission authorization ‚Äî auth only, not authz.

## Success criteria

- `prisma migrate` creates the 4 tables in Neon.
- `curl` signup returns a session cookie; a `User` + `Account` row appear.
- `curl` login with the same credentials succeeds; wrong password fails.
- Frontend form completes a signup and `useSession()` shows the user.
- Google/GitHub button redirects to the provider and returns an authenticated
  session.
